CloudWatch エラー監視ガイド — サブスクリプションフィルター vs メトリクスアラーム

このドキュメントでは、terraform/modules/app-infrastructure/cloudwatch.tf で併用している CloudWatch ログサブスクリプションフィルター と CloudWatch メトリクスアラーム の役割の違い、両方を併用する意義、および backend 側で整備すべき共通エラー処理についてまとめる。

TL;DR

  • 両者は 重複ではなく補完関係。片方では取れない情報・ケースをもう片方がカバーする。
  • メトリクスアラーム = 「Lambda 実行が落ちたかどうか」の検知（件数ベース、内容は含まれない）
  • サブスクリプションフィルター = 「何が起きたか」を通知 Lambda で整形して Slack/メールに送る（ログ本文ベース）
  • サブスクリプションフィルターを使うなら、backend 側で 構造化ログによる共通エラー処理 を入れて誤検知を防ぐべき。

1. 両者の役割の違い

1.1 メトリクスアラーム（aws_cloudwatch_metric_alarm）

対象メトリクス: AWS/Lambda 名前空間の Errors / Throttles など。

  • 検知対象: Lambda 実行そのものが失敗したケース
    • 未処理例外（throw されたまま関数を抜けた）
    • タイムアウト
    • Out Of Memory（メモリ不足による強制終了）
    • 初期化失敗（コールドスタート時の import エラー等）
    • Throttles は同時実行数上限による拒否
  • 取れる情報: 発生回数のみ（エラー内容・スタックトレースは含まれない）
  • 通知内容: 「1 分間にエラーが N 件発生した」という件数情報のみ
  • 特徴: しきい値・評価期間ベースで静かに・確実に検知できる

1.2 ログサブスクリプションフィルター（aws_cloudwatch_log_subscription_filter）

対象: CloudWatch Logs に出力されたログ行をパターンマッチ。

  • 検知対象: ログ本文に特定のパターン（例: ERROR / CRITICAL）を含む行
    • アプリケーションコードで logger.error(...) 等で出力されたログ
    • try/catch で捕捉して「500 を返しつつエラーログだけ出した」ケースも拾える
  • 取れる情報: ログ本文そのもの（タイムスタンプ、メッセージ、スタックトレース、リクエスト ID 等）
  • 通知内容: 通知 Lambda で整形すれば Slack/メールに エラー詳細 を含められる
  • 特徴: 「Lambda としては成功（200 OK）だが業務的にはエラー」のケースも捉えられる

1.3 比較表

観点 メトリクスアラーム サブスクリプションフィルター
検知対象Lambda 実行自体の失敗（未処理例外・タイムアウト・OOM）ログ本文に特定パターンを含む行
取れる情報発生回数のみログ本文・スタックトレース
try/catch で握り潰したエラー検知できない（Lambda は成功扱い）検知できる（ログに ERROR が出ていれば）
Lambda プラットフォーム起因の失敗検知できる検知しづらい（タイムアウト時に何もログが出ないケースがある）
通知に含まれる情報量件数のみ（CloudWatch を開かないと内容不明）整形次第でエラー詳細まで含められる
誤検知のリスク低い（公式メトリクスなので確実）中〜高（ログ本文のパターンマッチ次第）

1.4 片方だけにすると何が落ちるか

メトリクスアラームだけの場合:

  • 通知が「エラーが 1 件発生しました」で終わり、内容を見るには毎回 CloudWatch Logs を開く必要がある
  • アプリ側で捕捉してログ出力だけ行ったエラー（例: 外部 API 呼び出し失敗を 500 ではなく 200 + エラーフラグで返す等）が検知できない

サブスクリプションフィルターだけの場合:

  • Lambda がタイムアウト・OOM で強制終了した場合、ログに何も出ないケースがある
  • スロットリング（Throttles）はログには現れない（AWS 側で拒否されるため）
  • 結果として 「落ちたのに通知が来ない」 事故が起きうる

このプロジェクトで両方を併用しているのは、上記の片手落ちを防ぐため。

2. サブスクリプションフィルターを使うなら共通エラー処理が必須

2.1 現状の問題点

cloudwatch.tf の filter_pattern は以下のようになっている:

filter_pattern = "?ERROR ?CRITICAL"

これは「ログ本文に ERROR または CRITICAL という文字列を含む行」にマッチする単純なテキストフィルタ。以下の問題がある:

  • ライブラリや依存パッケージが情報ログに ERROR という単語を含めていると誤発火する
    • 例: "Retrying after 500 ERROR response..." のようなリトライ情報ログ
  • 開発者が console.log("no error") のように書いても ERROR を含むため誤発火する可能性がある
  • 通知 Lambda 側でログ内容を構造的にパースしづらい（プレーンテキストのため）

2.2 推奨される対策: 構造化ログ + JSON フィルタ

backend 側で以下を統一する:

  1. ログを JSON で出力 する（構造化ログ）
  2. エラー時は必ず level / message / requestId / stack 等のフィールドを持つ形で出す
  3. Hono なら app.onError(...) で一元的にエラーをハンドル

Terraform 側で filter_pattern を JSON フィルタに変える:

filter_pattern = "{ $.level = \"ERROR\" || $.level = \"CRITICAL\" }"

これにより:

  • 誤検知が激減する（level フィールドが ERROR の行だけが対象）
  • 通知 Lambda で JSON.parse(logEvent.message) すれば message / requestId / stack を構造的に取り出せる
  • Slack/メールに整形された読みやすい通知を送れる

2.3 Hono での共通エラーハンドリングの例

// backend/src/index.ts などに追加
app.onError((err, c) => {
  const requestId = c.req.header("x-request-id") ?? crypto.randomUUID();

  console.error(JSON.stringify({
    level: "ERROR",
    message: err.message,
    stack: err.stack,
    requestId,
    path: c.req.path,
    method: c.req.method,
    timestamp: new Date().toISOString(),
  }));

  return c.json({ error: "Internal Server Error", requestId }, 500);
});

ポイント:

  • 例外がここに集約されるため、「どこかでログ形式がバラバラ」になる問題を防げる
  • requestId を返すことで、ユーザーからの問い合わせ時にログ検索が容易になる
  • level: "ERROR" を必ず含めることで、サブスクリプションフィルターの JSON 条件に合致する

2.4 業務エラーと予期しないエラーを区別する

全てのエラーを ERROR レベルで通知すると、バリデーション失敗等のユーザ起因のエラーまで通知されてノイズになる。

ケース ログレベル 通知対象
バリデーションエラー（400）・認証エラー（401/403）WARN / INFO通知しない
業務ロジック上の想定済みエラーWARN通知しない
DB 接続失敗・外部 API 呼び出し失敗等の予期しないエラー（500）ERROR通知する
データ破損・セキュリティインシデント等の重大なエラーCRITICAL通知する

onError ハンドラで、HTTPException のステータスコードで分岐するなどして区別するとよい。

3. まとめ

  • メトリクスアラームとサブスクリプションフィルターは役割が違い、併用が推奨される
  • メトリクスアラームは「Lambda が落ちた」ことを確実に検知
  • サブスクリプションフィルターは「何が起きたか」を通知に含めるために使う
  • サブスクリプションフィルターを有効活用するには、backend 側で Hono の onError を使った共通エラーハンドラ + 構造化ログ（JSON） を整備し、Terraform の filter_pattern を JSON フィルタに変更するのが望ましい
  • 業務エラーと予期しないエラーはログレベルで区別し、通知ノイズを減らす
